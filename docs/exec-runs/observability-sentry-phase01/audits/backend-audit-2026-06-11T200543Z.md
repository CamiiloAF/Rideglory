## Resumen
La superficie compartida (`rideglory-common-lib/src/observability/`), el lado gateway
(serializer, CLS, header `x-trace-id`, interceptor de logging, `traceId` en error body)
y los guardrails de scope (sin Sentry, contracts intactos, http-logger eliminado) están
bien construidos y sus tests pasan. PERO el deliverable central de la fase —correlación
gateway→MS con el mismo `traceId` en los logs del microservicio (AC-1)— **no funciona y
además rompe el manejo de RPC**.

## Hallazgo crítico — CLS sin contexto en los microservicios

En cada MS (`users-ms`, `events-ms`, `vehicles-ms`, `maintenances-ms`, `notifications-ms`)
`app.module.ts` configura:

```ts
ClsModule.forRoot({ global: true, middleware: { mount: false } })
```

No se monta `ClsGuard` ni `ClsInterceptor` (`guard.mount` / `interceptor.mount` quedan en
`false` por defecto). Verificado en la fuente de nestjs-cls 6.2.1
(`dist/src/lib/cls-module/cls-root.module.js` líneas 166–181): el guard/interceptor que
inicializa el `AsyncLocalStorage` solo se registra cuando `mount` es `true`. En transporte
TCP no se establece ningún contexto CLS automáticamente.

Consecuencias verificadas en runtime (probe sobre `ClsServiceManager.getClsService()`
fuera de contexto, jest, gateway):
- `cls.get('traceId')` → `undefined`. El `mixin` de pino en `pinoHttpOptions(..., () =>
  cls.get('traceId'))` devuelve `{}`. **El `traceId` NUNCA aparece en las líneas de log de
  los MS → AC-1 incumplido.**
- `cls.set('traceId', traceId)` → **lanza** `Error: "Cannot set the key 'traceId'. No CLS
  context available..."`. Como el `TracingSerializer` del gateway ahora inyecta `traceId`
  en casi todos los envelopes, `ClsRpcInterceptor.intercept` ejecutará `cls.set` con
  `traceId` presente → **excepción en cada RPC trazado** = regresión funcional que rompe la
  API real (los tests no lo detectan porque no hay test del path MS).

## Cobertura de pruebas insuficiente para el path nuevo

- `ClsRpcInterceptor` no tiene NINGÚN test (verificado: ningún spec lo importa).
- No hay test de integración gateway→MS que pruebe AC-1 (mismo `traceId` en log de gateway
  y de MS). El e2e solo usa `HealthModule` con módulo mínimo en el gateway; nunca ejercita
  `TracingDeserializer` + `ClsRpcInterceptor` dentro de un contexto RPC.
- No hay humo de arranque ×6 (AC-11). Un arranque real habría expuesto el throw del
  interceptor en el primer RPC.

Esto contradice el criterio de auditoría "pruebas que fallarían sin el cambio" para el
guardrail backend más importante de la fase: el path que está roto no está cubierto.

## Cambios requeridos
1. Establecer contexto CLS en los MS para RPC: en cada `app.module.ts` usar
   `ClsModule.forRoot({ global: true, interceptor: { mount: true, setup: ... } })`
   (o `guard: { mount: true }`), y sembrar `traceId` desde `data._meta` EN el setup del
   inicializador CLS — no desde un APP_INTERCEPTOR aparte que corre fuera del contexto. El
   `ClsRpcInterceptor` actual no puede llamar `cls.set` porque no hay contexto activo; o se
   integra el seed dentro del `setup` del `ClsInterceptor` de nestjs-cls, o se envuelve el
   handler en `cls.run(...)`.
2. Garantizar el orden: el inicializador CLS debe abrir el contexto ANTES de que corra el
   interceptor que siembra `traceId` y antes de que pino lea el mixin.
3. Añadir test de integración gateway→MS (o al menos un test del MS) que verifique que una
   línea de log del MS contiene el `traceId` recibido por `_meta` — un test que falle con la
   implementación actual.
4. Añadir spec de `ClsRpcInterceptor` (incluyendo el caso de contexto activo) y humo de
   arranque para los 6 servicios (AC-11).
