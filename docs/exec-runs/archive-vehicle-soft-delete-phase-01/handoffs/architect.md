## Decisiones

| # | Decisión | Razón |
|---|----------|-------|
| 1 | `isDeleted Boolean @default(false)` en `Vehicle` (vehicles-ms Prisma schema) | Aditivo, sin romper filas existentes. Campo interno; nunca se expone en respuestas HTTP. |
| 2 | `softDeleteVehicle(vehicleId, ownerId)` en `VehiclesService` — transacción única | Ownership check + soft-delete + promoción de main en un solo `$transaction` para consistencia atómica. |
| 3 | `@MessagePattern('softDeleteVehicle')` en `VehiclesController` (vehicles-ms) — nuevo patrón | Separa semánticamente el soft-delete del hard-delete existente (`hardDeleteVehicle`). El hard-delete se conserva intacto. |
| 4 | Nuevo endpoint `DELETE /api/vehicles/my/:vehicleId` en api-gateway — autenticado, orquesta maintenances-ms → vehicles-ms | Ownership de Firebase token, no del body. Declarado **antes** de `@Delete(':id')` y antes de los handlers `soat`/`tecnomecanica` para evitar que NestJS interprete "my" como un UUID param. |
| 5 | `findByOwnerId` y `findMainVehicleByOwnerId` añaden `isDeleted: false, isArchived: false` al `where` | Los garajes solo muestran vehículos activos. |
| 6 | `findByIdOrNull` / `getVehicleById` — sin filtro `isDeleted` | Snapshots históricos de events-ms deben seguir accesibles aunque el vehículo esté eliminado. Guardrail crítico de regresión. |
| 7 | `create()` corrige conteo: `where: { ownerId, isArchived: false, isDeleted: false }` | Bugfix: el conteo actual ignora archivados/eliminados, causando que un vehículo nuevo no se marque `isMainVehicle: true` cuando todos los previos están archivados/eliminados. |
| 8 | Criterio de promoción canónico: `findFirst({ where: { ownerId, isArchived: false, isDeleted: false }, orderBy: { createdAt: 'desc' } })` | Coincide con la lógica existente del `remove()` hard-delete pero añade filtro `isDeleted`. Documentado aquí para que Fase 3 Flutter lo replique. |
| 9 | Tests Jest en `vehicles.service.spec.ts` (crear si no existe) | `soat.service.spec.ts` y `tecnomecanica.service.spec.ts` ya existen como referencia de patrón de prueba. |
| 10 | No se modifican `rideglory-contracts` ni la app Flutter | Sin DTOs nuevos; esta fase es backend-only. |

## Contratos (rideglory-api)

### Nuevo endpoint — api-gateway

| Campo | Valor |
|-------|-------|
| Method | `DELETE` |
| Path | `/api/vehicles/my/:vehicleId` |
| Auth | Firebase ID token (header `Authorization: Bearer <token>`) — owner derivado de `getAuthenticatedUser(request).id` |
| Path param | `vehicleId` — UUID válido (ParseUUIDPipe) |
| Request body | ninguno |
| Success | `200 { message: "Vehicle deleted successfully", status: 200 }` |
| 403 | `{ message: "Vehicle not found or does not belong to this owner", statusCode: 403 }` |
| 404 | `{ message: "Vehicle with id <uuid> not found", statusCode: 404 }` |
| 502 | maintenances-ms timeout / fallo — vehículo NO se soft-delete (consistencia conservadora) |

**Orden de orquestación:**
1. `softDeleteMaintenancesByVehicleId` → maintenances-ms (timeout 15s, igual que `hard-delete`)
2. `softDeleteVehicle` → vehicles-ms (solo si paso 1 resuelve sin error)

### Nuevo MessagePattern — vehicles-ms

```
Pattern: 'softDeleteVehicle'
Payload: { vehicleId: string; ownerId: string }
Success: Vehicle (la fila con isDeleted: true)
RpcException 403: ownership violation
RpcException 404: vehicleId no encontrado
```

### MessagePattern existente — maintenances-ms (sin cambios)

```
Pattern: 'softDeleteMaintenancesByVehicleId'
Payload: { vehicleId: string }
```
Ya implementado y funcional. No requiere modificación.

## Riesgos

| Riesgo | Severidad | Mitigación |
|--------|-----------|------------|
| `findByIdOrNull` recibe filtro `isDeleted` accidentalmente | ALTO | Guardrail explícito en code review: el `where` de `findByIdOrNull` debe tener SOLO `{ id }`. Verificar que ningún cambio en el método lo altere. |
| Orden de rutas en api-gateway: NestJS interpreta "my" como UUID | MED | `@Delete('my/:vehicleId')` DEBE declararse antes de `@Delete(':id')` y antes de cualquier `@Delete(':vehicleId/...')`. Verificar en code review. |
| `isDeleted` expuesto en respuesta HTTP de `GET /api/vehicles/my` | MED | Prisma retorna todos los campos del modelo; verificar que el campo no aparezca en la respuesta pública. Si es necesario, añadir select explícito que lo excluya. |
| Migración contiene DROP o sentencias destructivas | MED | Inspeccionar `migration.sql` generado antes de commitear. Rechazar si contiene cualquier sentencia no-`ADD COLUMN`. |
| `hard-delete/:id` modificado accidentalmente | MED | El endpoint `hard-delete/:id` no debe tocarse. Verificar que el `hardDeleteVehicle` MessagePattern siga siendo un hard-delete físico. |
| Transacción `softDeleteVehicle` falla a mitad (soft-delete OK, promoción de main KO) | LOW | Wrapped en `$transaction` — rollback automático. |

## Superficie de regresión

- `GET /api/vehicles/my` — debe excluir `isDeleted: true` y `isArchived: true`
- `GET /api/vehicles/:id` (findOneVehicle / getVehicleById) — debe seguir funcionando para vehículos soft-deleted (events-ms snapshots)
- `DELETE /api/vehicles/hard-delete/:id` — debe seguir haciendo hard-delete físico sin cambios
- `PUT /api/vehicles/my/:vehicleId/main` — no afectado, pero debe verificarse que sigue funcionando
- Creación de vehículo cuando el único previo está archivado/eliminado — debe marcarse como `isMainVehicle: true`
- `dart analyze` y `flutter test` — pasan en verde (sin cambios Flutter)
