## Open questions resolved

### Decision 1 — `VehicleSummaryModel?` deserialization in `EventRegistrationDto`

**Resolution:** Type the constructor parameter as `VehicleSummaryDto?` (the DTO subclass) using `super.vehicleSummary`. Because `VehicleSummaryDto extends VehicleSummaryModel`, the parent's `vehicleSummary` field (`VehicleSummaryModel?`) accepts the subtype assignment, and `json_serializable` will generate `VehicleSummaryDto.fromJson(...)` for the nested object automatically (it inspects the constructor parameter type, not the field type on the parent class). No `@JsonKey(fromJson: ...)` converter is needed.

**Pattern (in `event_registration_dto.dart`):**
```dart
class EventRegistrationDto extends EventRegistrationModel {
  const EventRegistrationDto({
    ...
    super.vehicleId,
    VehicleSummaryDto? super.vehicleSummary, // DTO type on constructor
    ...
  });
}
```
If Dart's `super` parameter syntax rejects narrowing the type at the call site (it generally does for covariant parameters in subclass constructors via the explicit type annotation form above), the fallback is to declare the parameter independently:
```dart
const EventRegistrationDto({
  ...
  this.vehicleSummaryDto,
  ...
}) : super(vehicleSummary: vehicleSummaryDto);

final VehicleSummaryDto? vehicleSummaryDto;
```
**Frontend must try the `super` form first; fall back to the field-shadow form only if codegen fails.** The simpler form is preferred for readability.

`EventModel`/`EventDto` does not nest other DTOs, so this is the only place the pattern is exercised.

### Decision 2 — `_buildCreateBody` in `maintenance_repository_impl.dart`

**Resolution:** **Leave `_buildCreateBody` as-is** for this iteration. It is **not** a 1:1 model-serialization payload — it adds two fields that are not on `MaintenanceModel` (`nextKmInterval` is a request-only hint, and `mode` is hand-mapped to API strings). Forcing it through `maintenance.toJson()` would require either (a) inventing a request DTO or (b) a post-`toJson()` mutation, both of which expand scope.

**Memory rule citation:** the `feedback_dto_toJson.md` rule ("API write payloads must use DTO `.toJson()`; never manually construct `Map<String, dynamic>`") technically applies — but the cleanest compliant solution is a dedicated request DTO (`CreateMaintenanceRequestDto`), which is out-of-scope for a refactor-only iteration. Capture as follow-up tech-debt in the QA handoff.

**However**, line 182 — `MaintenanceDto.fromModel(maintenance).toJson()` — DOES fit the rule and MUST become `maintenance.toJson()` (via the new `MaintenanceModelExtension`).

### Decision 3 — `SoatDto.expiryDate` nullability

**Resolution:** Make DTO `expiryDate` **non-nullable** (`required super.expiryDate`), matching the model.

**Rationale:**
- `SoatModel.expiryDate` is `required DateTime` (non-nullable).
- The current DTO permits null and falls back to `DateTime.now()` in `toModel()`. This was a defensive guard; the backend contract guarantees a non-null `expiryDate` whenever a SOAT row exists (no SOAT → 404, handled separately in `getSoat`).
- Falling back to `DateTime.now()` silently masks bad data. Stricter typing surfaces it via deserialization error, which is the right outcome.

**Pattern (in `soat/data/dto/soat_dto.dart`):**
```dart
class SoatDto extends SoatModel {
  const SoatDto({
    required super.id,
    required super.vehicleId,
    super.policyNumber,
    super.startDate,
    required super.expiryDate, // non-nullable
    super.insurer,
    super.documentUrl,
    super.createdAt,
    super.updatedAt,
  });

  factory SoatDto.fromJson(Map<String, dynamic> json) => _$SoatDtoFromJson(json);
  Map<String, dynamic> toJson() => _$SoatDtoToJson(this);
}

// Preserve verbatim — used by save flow.
extension SoatModelToRequest on SoatModel {
  Map<String, dynamic> toRequestJson() { /* unchanged */ }
}
```
`apiJsonDateTimeConverters` already includes a required-DateTime converter, so codegen will pick the correct one.

If the API ever returns a null `expiryDate`, this becomes a deserialization error — the right loud failure. **Frontend must add a brief code comment noting that null is treated as a backend contract violation.**

## Risk register

1. **JSON wire-format drift (HIGH).** Any missing `@JsonKey` on remapped names (esp. `createdAt`/`updatedAt` → `createdDate`/`updatedDate` in `MaintenanceDto`) breaks serialization silently. **Mitigation:** QA must diff a captured `toJson()` output before/after on a real instance, or add a unit-test asserting key set.
2. **`super` parameter codegen with nested DTOs (MEDIUM).** `json_serializable` parameter-type detection on `super`-prefixed params is robust but not universally tested. **Mitigation:** Frontend uses the field-shadow fallback (decision 1) if first pass fails; build_runner errors will surface immediately.
3. **`SoatDto.expiryDate` runtime exception (MEDIUM).** If the backend ever returns null, deserialization throws. **Mitigation:** Document this contract assumption in a code comment on the DTO; QA verifies the API never returns null on the happy-path call we already exercise.
4. **`_buildCreateBody` rule deviation (LOW).** Memory rule says use DTO `.toJson()` for API writes; we leave this body manual. **Mitigation:** Log as follow-up tech-debt in QA handoff so it gets ticketed, not forgotten.
5. **Orphaned `.g.dart` files (LOW).** If a `.g.dart` is left from an earlier shape, the file may have stale members. **Mitigation:** Frontend runs `dart run build_runner build --delete-conflicting-outputs` (the `--delete-conflicting-outputs` flag is mandatory).
6. **Test breakage on `.toModel()` removal (LOW).** Some test may currently call `dto.toModel()`. **Mitigation:** QA grep before running tests.
