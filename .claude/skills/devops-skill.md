## Domain context

Rideglory is a Flutter mobile app targeting Android and iOS. The DevOps role delivers and maintains the GitHub Actions CI/CD pipeline. The pipeline must be automated from Iteration 3 onward so that every PR is gated by `dart analyze` + `flutter test` without manual intervention.

DevOps track runs **parallel to Iteration 2** and delivers `.github/workflows/flutter-ci.yml`.

---

## Scope boundaries

**In scope (DevOps track):**
- GitHub Actions workflow: `dart analyze` + `flutter test` on every push to `iter-N` branches and every PR to `main`
- APK build job triggered on version tags (e.g., `v*`)
- Environment variable injection from GitHub Actions secrets
- Firebase config file injection from CI secrets (`google-services.json`, `GoogleService-Info.plist`)

**Out of scope:**
- IPA build + App Store submission (deferred — requires distribution target confirmation)
- Fastlane or other distribution automation
- Test device farms (emulator in CI is sufficient)
- Monitoring, alerting, or observability beyond CI status

---

## Infrastructure constraints

- **No secrets in source.** `.env.example` contains placeholders only. Real values live in GitHub Actions secrets and are injected at build time.
- **Firebase config files not committed.** `android/app/google-services.json` and `ios/Runner/GoogleService-Info.plist` are injected from CI secrets. Example files (`.example`) may be committed.
- **Minimum SDK targets:** Android API 21, iOS 13. CI must use a Flutter version that supports these.
- **Branch pattern:** Iteration branches are `iter-1`, `iter-2`, `iter-3a`, `iter-3b`, `iter-4`, `iter-5`, `iter-6a`, `iter-6b`. CI triggers on `iter-*` pattern.

## CI pipeline requirements

### Trigger
```yaml
on:
  push:
    branches: ['iter-*', 'main']
  pull_request:
    branches: ['main']
```

### Jobs

**1. analyze-and-test (required gate)**
Steps:
1. Checkout code
2. Setup Flutter (stable channel, version pinned from `pubspec.yaml`)
3. Inject `.env` from GitHub Actions secret `ENV_FILE`
4. Inject `google-services.json` from secret `GOOGLE_SERVICES_JSON`
5. Inject `GoogleService-Info.plist` from secret `GOOGLE_SERVICE_INFO_PLIST`
6. `flutter pub get`
7. `dart run build_runner build --delete-conflicting-outputs`
8. `dart analyze` — fail on any violation
9. `flutter test` — fail on any test failure

**2. build-apk (optional, on version tags)**
```yaml
on:
  push:
    tags: ['v*']
```
Steps: same setup → `flutter build apk --release` → upload artifact

### Key acceptance pattern
- `analyze-and-test` must be a **required status check** on PRs to `main`
- No bypass of CI gate
- APK artifact retained for 30 days after tag build

---

## Key acceptance patterns (DevOps track definition of done)

1. `.github/workflows/flutter-ci.yml` exists and is syntactically valid.
2. `analyze-and-test` job runs on every push to `iter-*` and every PR to `main`.
3. A test push with a deliberate `dart analyze` violation fails the job (red X).
4. A test push with a deliberate failing test fails the job (red X).
5. Secrets are injected correctly — `dart run build_runner build` completes without "missing env var" errors.
6. APK build job runs on a `v0.0.1` test tag and produces an artifact.
7. `DEPLOY.md` exists documenting: how to create a release tag, what the CI does, how to update secrets.
