## Domain context

Rideglory es una app para riders y organizadores de eventos en Colombia. Contexto de uso crítico para UX:

**Condiciones de uso extremas:**
- Al aire libre, luz solar directa → contrastes altos, dark mode obligatorio.
- Con guantes en la moto → touch targets generosos (≥ 48px para acciones críticas).
- Velocidad → información jerárquica, escaneable, sin sobrecarga cognitiva.
- Emergencias (SOS) → flujo sin fricción, cero ambigüedad.

**Personas:**
- **Rider** — navega eventos, rastrea rodadas, gestiona su garaje. Quiere info rápida y accesible.
- **Organizador** — gestiona ciclo del evento, revisa inscripciones, monitorea riders en mapa. Necesita control, no solo lectura.

**Copy (español colombiano):**
- Sentence case en botones: `'Iniciar sesión'`, no `'INICIAR SESIÓN'`.
- Tono directo y funcional — herramienta de seguridad y logística.
- Errores: claros, accionables, en español plano.

- `#F4F4F5` sobre `#0A0A0A` → ratio ~17:1 ✓

- Blanco `#FFFFFF` sobre `#f98c1f` → ratio ~2.9:1 ✗ **BLOQUEANTE**

## Patrones UX aprobados (no re-discutir)

Estos patrones ya fueron validados en iteraciones anteriores:

| Patrón | Decisión |
|--------|---------|
| Carga | Skeleton/shimmer — nunca `CircularProgressIndicator` |
| Estado vacío | `EmptyStateWidget` — nunca pantalla en blanco |
| Error | Banner o card con botón "Reintentar" — nunca solo texto rojo |
| Formularios multi-paso | Indicador "Paso N de M" visible |
| Overlays de alerta | Banner top-anchor, mapa interactivo debajo (no modal bloqueante) |
| Upload | 4 fases visuales: selección → progreso → procesamiento → confirmación |
| Switches | Solo `AppSwitch` (pill) / `AppSwitchTile` — nunca Material `Switch` o `FormBuilderSwitch` |
| Botones | Solo `AppButton` / `AppTextButton` — nunca `ElevatedButton` / `TextButton` |
| Texto sobre primario | Solo texto/iconos oscuros (`#0D0D0F`) — nunca blanco sobre naranja |

## Reglas Pencil para el revisor

- Usar `get_editor_state(include_schema: true)` antes de cualquier otra operación.
- `batch_get` para leer contenido del frame; `get_screenshot` para captura visual.
- Fondos oscuros a veces retornan blanco en `get_screenshot` → usar `snapshot_layout` en ese caso.
- `export_nodes`: un nodo por llamada; frames altos (≥ 1445px) no exportan bien a >1x.
- El revisor NO usa `batch_design` — solo lee, nunca edita el .pen.
- Si Pencil MCP no responde (Pencil desktop no activo), documentarlo en el reporte y hacer revisión parcial basada en handoffs de texto.

## Gotchas y learnings

- **Contraste en dark mode**: los diseñadores tienden a usar `color-text-secondary` (`#71717A`) en subtítulos que en realidad necesitan `color-text-primary` si son funcionales. Verificar ratio real.
- **Targets en mapa**: los `PointAnnotation` de Mapbox tienen touch area limitada — revisar que los controles del mapa tienen suficiente padding.
- **Bottom sheets**: el `handle` del sheet no cuenta como touch target — el primer elemento interactivo debe ser ≥ 44px.
- **Modo organizador vs rider**: muchos frames tienen variantes condicionales. El revisor debe pedir ambas variantes si solo hay una en Pencil.

## Scope boundaries

**IN scope para el revisor:**
- Todos los frames de Pencil que corresponden a pantallas con UI nueva o modificada.
- Estados UX: idle, loading, success, error, vacío.
- Accesibilidad visual (contraste, touch targets, labels).
- Consistencia con el design system y los patrones aprobados.

**OUT of scope:**
- Implementación Flutter (eso es Tech Lead).
- Performance y animaciones (a menos que el diseño las especifique y sean inapropiadas).
- Lógica de negocio — solo UX.
