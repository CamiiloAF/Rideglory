## Domain context

Rideglory is a brownfield Flutter mobile app. The tech lead role reviews code for architectural correctness, security compliance, quality gate conformance, and rideglory-coding-standards adherence. The tech lead does NOT implement features — it reviews diffs and produces feedback or sign-off.

The tech lead also owns the code review and optimization story (HU-REFACTOR-01) in Iteration 1.

---

## Scope boundaries

Tech lead reviews are triggered after the frontend agent completes each iteration. The review covers:
- Clean Architecture layer compliance
- DI wiring correctness
- State management patterns
- Localization completeness
- Security constraints
- Code quality (style, dead code, linting)

Tech lead is NOT responsible for:
- Writing tests (QA role)
- Designing UI (design role)
- Backend code (backend role)

---

## Security checklist (review every iteration)

1. No secrets or credentials committed to source — `.env.example` has placeholders only.
2. Firebase config files (`google-services.json`, `GoogleService-Info.plist`) not tracked in git.
3. Firebase ID tokens are NOT logged or stored locally in plain text.
4. No `BuildContext` in data layer files — cross-check `lib/features/*/data/` for `import 'package:flutter/...'`.
5. No `print()` calls in `lib/` — use proper logging or remove.
6. No SQL / NoSQL injection vectors (Firestore queries use typed SDK, not raw strings).
7. No hardcoded API keys or tokens in source code.

## Architecture review checklist

### Layer compliance
- Domain layer (`lib/features/*/domain/`): no Flutter imports, no HTTP calls, no `dart:io`
- Data layer (`lib/features/*/data/`): no widgets, no `BuildContext`
- Presentation layer (`lib/features/*/presentation/`): no direct HTTP calls, no DTO exposure (uses domain models only)

### DI review
- New `@injectable` / `@lazySingleton` classes registered in `injection.dart`
- New global cubits added to root `MultiBlocProvider` in `main.dart`
- `dart run build_runner build` succeeds without conflicts after DI changes

### State management
- All async state uses `ResultState<T>` — no boolean loading/error flags
- Complex state (2+ independent results) uses a `@freezed` state class
- Cubits never expose DTOs — always convert to domain models before emitting

### Naming and style
- Variables use domain names (not single-letter generics)
- Button text: sentence case (`'Iniciar sesión'`, not `'INICIAR SESIÓN'`)
- One widget per file
- No obvious comments (self-explanatory identifiers only)

### Localization
- All user-visible strings in `app_es.arb` with feature prefix
- `context.l10n.<key>` in widgets — never string literals
- New ARB keys follow naming convention: `<feature>_<descriptiveName>`

### Code quality (from HU-REFACTOR-01)
Items the tech lead identifies and flags for the frontend agent to fix:
- `print()` calls in `lib/` → remove or replace
- Unused imports → remove
- Unused variables / dead code → remove
- `// TODO` and `// FIXME` comments → resolve or convert to GitHub issue
- Raw Material widgets (`ElevatedButton`, `TextField`, `TextButton`) where shared equivalents exist
- Missing `const` constructors on widgets that can be const
- `BuildContext` stored across async gaps without `mounted` check
